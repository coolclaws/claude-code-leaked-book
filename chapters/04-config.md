# 第 4 章 配置系统

> "Configuration is the silent contract between software and its environment."
> —— Adam Jacob

一个成熟的 CLI 工具需要同时服务个人开发者、团队项目和企业 IT 管理员。这三类用户对配置的诉求截然不同：个人关注偏好，团队关注一致性，企业关注合规与管控。Claude Code 的配置系统通过多层合并策略，在一套统一的机制下满足了这三种需求。

## 4.1 配置层次模型

Claude Code 的配置遵循"逐层覆盖"原则，低优先级的配置可以被高优先级的配置覆盖。完整的优先级从低到高为：

```
优先级（低 -> 高）:

+-----------------------------+
|  1. 内置默认值 (settings.ts)|  代码中的硬编码默认值
+-----------------------------+
          |  覆盖
+-----------------------------+
|  2. 全局用户配置 (~/.claude/)|  用户个人偏好
+-----------------------------+
          |  覆盖
+-----------------------------+
|  3. 项目配置 (.claude/)     |  项目级别的约定
+-----------------------------+
          |  覆盖
+-----------------------------+
|  4. MDM 企业托管配置        |  IT 管理员的强制策略
+-----------------------------+
          |  覆盖
+-----------------------------+
|  5. 环境变量                |  运行时动态覆盖
+-----------------------------+
          |  覆盖
+-----------------------------+
|  6. CLI 参数                |  当次调用的临时覆盖
+-----------------------------+
```

这套层次关系与 Git 的配置模型（system -> global -> local）类似，但增加了 MDM 企业管理层和环境变量层，以适应更复杂的部署场景。

## 4.2 内置默认值：settings.ts

配置的基底是 `settings.ts` 中定义的默认值：

```typescript
// src/utils/settings/settings.ts
// 定义所有配置项的默认值
// 每个配置项都有类型定义和文档注释
// 例如：模型选择、超时时间、输出格式等
```

默认值的设计遵循"开箱即用"原则——用户在不做任何配置的情况下，Claude Code 应该以合理的行为运行。这要求每个默认值都经过仔细斟酌：太保守会让新用户觉得功能不足，太激进则可能在某些环境下产生问题。

## 4.3 全局与项目配置

用户配置分为两个作用域：

**全局配置** 存放在 `~/.claude/` 目录下，影响用户在所有项目中的行为。典型的全局配置包括 API 密钥存储路径、默认模型选择、UI 偏好等。

**项目配置** 存放在项目根目录的 `.claude/` 下，只影响当前项目。典型的项目配置包括允许的工具列表、自定义系统提示词、MCP 服务器声明等。项目配置通常会提交到版本控制系统，确保团队成员共享一致的行为。

配置加载由 `config.ts` 中的 `enableConfigs()` 函数驱动：

```typescript
// src/utils/config.ts
// enableConfigs() 负责：
// 1. 定位全局配置目录 (~/.claude/)
// 2. 定位项目配置目录 (.claude/)
// 3. 读取并解析 JSON/YAML 配置文件
// 4. 按优先级合并为最终配置对象
```

合并逻辑并非简单的浅拷贝覆盖。对于数组类型的配置项（如允许的工具列表），需要决定是替换还是追加；对于嵌套对象，需要递归合并。这些细节决定了配置系统在实际使用中是否符合直觉。

## 4.4 MDM 企业托管配置

企业环境中的配置管理是 Claude Code 区别于多数开源 CLI 工具的一个重要特性。MDM（Mobile Device Management）层允许 IT 管理员通过操作系统级别的管理工具下发强制配置。

MDM 配置的读取在启动序列中被提前触发（见第 3 章），其实现位于 `rawRead.ts`：

```typescript
// src/utils/settings/mdm/rawRead.ts
// startMdmRawRead() 根据操作系统选择不同的读取方式：
// - macOS: 通过 plutil 命令读取 plist 配置
// - Windows: 通过 reg query 读取注册表
// - Linux: 读取特定路径下的配置文件
```

MDM 读取被设计为子进程调用而非直接文件读取，原因在于操作系统的管理配置可能经过加密或存储在特殊位置（如 macOS 的 Managed Preferences），只有系统工具才能正确解析。

在优先级设计上，MDM 配置高于项目配置，这意味着企业管理员可以强制覆盖项目级设置。例如，管理员可以禁用某些工具或强制使用特定的 API 端点，无论项目配置如何声明。这种设计对于需要合规管控的企业环境至关重要。

远程托管配置通过 `remoteManagedSettings` 服务获取：

```typescript
// src/services/remoteManagedSettings/index.ts
// 提供从远程服务器获取企业托管配置的能力
// 与本地 MDM 配置互补，支持更动态的策略下发
```

## 4.5 环境变量与 CLI 参数

优先级最高的两层配置来自运行时：

**环境变量** 适用于 CI/CD 流水线或容器化部署等场景。例如，`ANTHROPIC_API_KEY` 可以直接设置 API 密钥，`CLAUDE_CODE_PROFILE_STARTUP` 可以开启启动性能分析。环境变量的命名惯例是 `CLAUDE_CODE_` 前缀加大写蛇形命名。

**CLI 参数** 是最终的覆盖手段。当用户在命令行中显式指定参数时，该参数拥有最高优先级。这保证了用户在任何情况下都能通过命令行临时改变行为，不受任何持久化配置的约束。

## 4.6 MCP 服务器配置的多作用域模型

MCP（Model Context Protocol）服务器的配置是整个配置系统中最复杂的部分，因为它需要支持多达七种作用域：

```typescript
// src/services/mcp/config.ts
// MCP 配置作用域：
// - local:      项目根目录 .mcp.json（不提交到版本控制）
// - user:       ~/.claude/mcp.json（用户全局）
// - project:    .claude/mcp.json（项目级，提交到版本控制）
// - dynamic:    运行时动态注册
// - enterprise: MDM 托管
// - claudeai:   Claude AI 平台下发
// - managed:    远程托管
```

`local` 和 `project` 的区别值得注意。`local` 对应 `.mcp.json` 文件，位于项目根目录但不应提交到版本控制（类似 `.env` 文件）；`project` 对应 `.claude/mcp.json`，设计为提交到版本控制的共享配置。这种分离让开发者可以在本地配置包含个人凭证的 MCP 服务器，同时通过 `project` 作用域共享团队公用的服务器声明。

`dynamic` 作用域允许在运行时通过代码注册 MCP 服务器，这对于需要根据上下文动态连接不同服务的场景非常有用。

`enterprise`、`claudeai` 和 `managed` 三个作用域分别对应不同来源的中心化管理：本地 MDM 策略、Claude AI 平台配置和远程托管服务。这三者的存在确保了企业和平台运营者可以在不修改用户本地文件的情况下控制 MCP 服务器的可用范围。

MCP 配置的合并顺序遵循与主配置相同的"低优先级被高优先级覆盖"原则，但增加了一个额外维度：**同名服务器的冲突解决**。当多个作用域声明了同名的 MCP 服务器时，高优先级作用域的声明会完全替换低优先级的声明，而非合并其参数。

## 4.7 配置的完整加载流程

将以上各层串联起来，配置加载的完整流程如下：

```
启动
  |
  +-- startMdmRawRead()  [子进程: plutil/reg query]
  |         |
  |    [后台读取 MDM 配置]
  |
  +-- 加载内置默认值 (settings.ts)
  |
  +-- 读取全局配置 (~/.claude/)
  |
  +-- 读取项目配置 (.claude/)
  |
  +-- 等待 MDM 数据就绪
  |   合并 MDM 配置
  |
  +-- 读取远程托管配置 (remoteManagedSettings)
  |
  +-- 应用环境变量覆盖
  |
  +-- 应用 CLI 参数覆盖
  |
  +-- 最终配置对象就绪
  |
  +-- 加载 MCP 配置 (七种作用域独立合并)
  |
  +-- 进入业务逻辑
```

注意 MDM 读取被提前到启动阶段并行执行（与模块加载重叠），但合并操作需要等待子进程完成。这是第 3 章中讨论的并行预取策略在配置系统中的具体体现。

## 本章小结

Claude Code 的配置系统通过六层优先级模型，在一套统一机制下同时服务个人偏好、团队约定和企业管控三种需求。内置默认值确保开箱即用，全局和项目配置提供灵活的持久化定制，MDM 层赋予企业管理员强制管控的能力，环境变量和 CLI 参数则保证了运行时的最终覆盖权。MCP 配置的七种作用域进一步展示了这套模型的扩展性——即使面对复杂的多来源服务器管理场景，同样的"逐层覆盖"原则依然适用。理解这套层次模型，是理解 Claude Code 在不同环境下行为差异的关键。
